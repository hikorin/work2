# ソフトウェア開発プロセス

新規プログラムの作成や機能追加を行う際は、必ず以下のステップを順守し、各工程の完了確認・報告を行いながら進行してください。各段階で関連するドキュメントやテストコードの作成が必要です。
※【例外ルール (Fast-Track Mode)】：ユーザーから「確認なしで一気に進めて」と指示された場合は、フェーズ間の待機（合意ステップ）を省略し、エージェントの自律的な判断で最終工程まで連続実行して構いません。

## モデル選定ポリシー

フェーズの性質に応じて、使用するAIモデルを以下の通り使い分けること。

| モデル | 役割 | 適用フェーズ |
|---|---|---|
| **Claude Opus 4.6 (Thinking)** 👑 | 最も深い思考・要求の本質的な分析が必要な工程 | 要求分析 |
| **Claude Sonnet 4.6 (Thinking)** 🧠 | 批判的分析・構造設計・レビューが必要な工程 | 各種レビュー、設計、テスト設計、コードレビュー |
| **Gemini 3.1 Pro** ⚡ | 実装・コマンド実行・対話・報告など実務的な工程 | 実装、テスト実施、ユーザー確認・報告 |

## 開発ステップ

1. **要求分析（ドキュメント作成）** 👑 [Claude Opus 4.6 (Thinking)]
   - ユーザーからの要望をもとに、要件定義書（`requirements.md`等）を作成する。
   - 要求の本質を深く理解し構造化する最重要フェーズのため、最上位モデル Opus を使用する。

2. **マルチエージェントによるレビュー** 🧠 [Claude Sonnet 4.6 (Thinking)]
   - 他の視点から要件定義書をレビューし、矛盾や抜け漏れがないか確認する。
   - 批判的・多角的分析が必要なため、Thinking モデルを使用する。

3. **ユーザによる確認** ⚡ [Gemini 3.1 Pro]
   - レビュー済みの要件定義書をユーザーに提示し、合意・承認を得る。
   - 対話・報告フェーズのため、Gemini 3.1 Pro を使用する。

4. **設計（ドキュメント作成）** 🧠 [Claude Sonnet 4.6 (Thinking)]
   - 要件定義に基づき、アーキテクチャ・データモデルなどの設計書（`design.md`等）を作成する。
   - 複雑なアーキテクチャ設計に深い思考が必要なため、Thinking モデルを使用する。

5. **マルチエージェントによる設計レビュー** 🧠 [Claude Sonnet 4.6 (Thinking)]
   - 設計内容が要件を満たしているか多角的にレビューする。
   - 批判的分析が必要なため、Thinking モデルを使用する。

6. **設計確定（フィックス）** ⚡ [Gemini 3.1 Pro]
   - 指摘事項を反映し、設計を最終的に確定（フィックス）する。
   - ドキュメント更新・整理フェーズのため、Gemini 3.1 Pro を使用する。

7. **テスト設計** 🧠 [Claude Sonnet 4.6 (Thinking)]
   - モジュール単体や機能レベルでのテストコード・テストケースの設計を行う。
   - 網羅的・論理的なテスト設計に深い思考が必要なため、Thinking モデルを使用する。

8. **マルチエージェントによるテスト設計レビュー** 🧠 [Claude Sonnet 4.6 (Thinking)]
   - テスト設計の網羅性をチェックし、不十分な箇所がないかレビューする。
   - 批判的分析が必要なため、Thinking モデルを使用する。

9. **マルチエージェントによる総括設計レビュー** 🧠 [Claude Sonnet 4.6 (Thinking)]
   - テスト容易性の観点も含め、システム全体の設計について最終の総括レビューを行う。
   - 全体を俯瞰した深い分析が必要なため、Thinking モデルを使用する。

10. **実装** ⚡ [Gemini 3.1 Pro]
    - 設計とテスト仕様に忠実に、実際のコーディング（本番コードとテストコード）を行う。
    - コーディング・ファイル操作の実務フェーズのため、Gemini 3.1 Pro を使用する。

11. **マルチエージェントによるコードレビュー** 🧠 [Claude Sonnet 4.6 (Thinking)]
    - 実装したコードの品質（可読性、バグの有無、セキュリティ）を自己・または別エージェントとしてレビューする。
    - バグ検出・セキュリティ分析に深い思考が必要なため、Thinking モデルを使用する。

12. **テスト実施** ⚡ [Gemini 3.1 Pro]
    - 実装した単体・機能テストを実行する。
    - コマンド実行・結果確認の実務フェーズのため、Gemini 3.1 Pro を使用する。

13. **テスト結果によるやり直し判定** 🧠 [Claude Sonnet 4.6 (Thinking)]
    - テスト不合格（NG）が発生した場合、原因を特定し、実装・テスト設計・設計の該当箇所まで遡って修正を行う（合格するまでループ）。
    - 根本原因分析（RCA）に深い思考が必要なため、Thinking モデルを使用する。

14. **結合テスト設計・レビュー** 🧠 [Claude Sonnet 4.6 (Thinking)]
    - 設計全体に基づく結合レベルのテストを設計し、問題なく動作するかテスト・レビューを行う。
    - 統合的な設計分析が必要なため、Thinking モデルを使用する。

15. **要求仕様のテスト設計・実施** ⚡ [Gemini 3.1 Pro]
    - 当初の「要求仕様」を完全に満たしているかを確認するための検証（システムテスト／受け入れテスト相当）を設計し、実施する。
    - テスト実行・結果確認フェーズのため、Gemini 3.1 Pro を使用する。

16. **最終判定・報告** ⚡ [Gemini 3.1 Pro]
    - 一連の要求要件、各種テストをすべてクリアしたか最終確認を行い、ユーザーに開発完了の報告をする。
    - ユーザーへの報告・対話フェーズのため、Gemini 3.1 Pro を使用する。

---

**運用ルール（Tips）**
- **エージェント向け指示**: この機能開発を開始する際はユーザーから「`/development_process` に従って〇〇を作って」といった形で指示を受けることができます。「開発フローに則って」などでも構いません。
- **ドキュメントの保存**: 各ステップで作成されるファイルは、リポジトリ内の適切な場所（例: `docs/` ディレクトリ等）へ随時保存してください。
- **モデル切り替えの注意**: フェーズ移行時にモデルを切り替える際は、引き継ぎ情報（直前のステップの成果物・判断内容）を次のモデルへ明示的に伝えること。
