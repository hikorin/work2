# 設計書: 納品・請求連動機能の強化とUIリファイン

## 1. 目的
- 請求書の削除を許可し、削除時に納品書のステータスを「未請求」に自動復元する。
- 納品書の編集・削除フローを改善し、請求ステータスに応じた適切な制御を行う。
- アプリ全体のボタンおよびUIパーツに角丸を適用し、モダンでプレミアムなデザインを実現する。

## 2. 変更内容

### 2.1. バックエンド (Python/FastAPI)
- **Invoice Router (`routers/invoice.py`)**:
  - `delete_invoice` エンドポイントが `models.Delivery` の `invoice_id` を確実に `None` に更新することを保証する。
- **Delivery Router (`routers/delivery.py`)**:
  - `update_delivery` / `delete_delivery` において、`invoice_id` が設定されている場合はエラー（400 Bad Request）を返し、「請求書を削除してから操作してください」といった具体的なメッセージを提供する。

### 2.2. フロントエンド (React/TypeScript)
- **スタイル定義 (`index.css`)**:
  - ボタンの角丸を `2px` から `24px`（カプセル型）または `12px` に変更。
  - 入力フィールド、カードの角丸も `8px` 程度に緩和し、全体的に柔らかい印象にする。
  - ホバー時のエフェクトを強調し、インタラクティブ性を高める。
- **納品書管理 (`components/DeliveryManager.tsx`)**:
  - 編集ボタン、削除ボタンのアイコンと配置を最適化。
  - すでに請求書に紐付いている場合、ボタンを無効化するか、警告を表示する。
- **請求書管理 (`components/InvoiceGenerator.tsx`)**:
  - 削除ボタン実行時の確認ダイアログのメッセージを、連動の仕組みが伝わる内容に更新。

### 2.3. データベース
- スキーマ変更なし（既存の `Deliveries.invoice_id` (FK) を利用）。

## 3. テスト計画
- **単体テスト**: 
  - 請求書削除時の納品書フラグ更新ロジックの検証。
- **E2Eテスト**:
  1. 納品書を作成する。
  2. 請求書を作成し、その納品書を含める。
  3. 納品書が「請求済み」になり、編集不可になることを確認。
  4. 請求書を削除する。
  5. 納品書が「未請求」に戻り、再度編集可能になることを確認。

## 4. デザイン指針（UIリファイン）
- **丸みの適用**:
  - `button`: `border-radius: 24px`
  - `card`: `border-radius: 12px`
  - `input/select`: `border-radius: 8px`
- **カラーアクセント**: 
  - 既存の「Kaeshi Precision」テーマ（紺・緑・赤）を維持しつつ、角丸と余白を調整してプレミアム感を演出する。
